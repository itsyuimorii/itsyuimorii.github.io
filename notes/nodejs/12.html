# 0. Using template engines with Express

# 1. EJS

# 1.1 Installation ejs (Embedded JavaScript templates)

$ npm install ejs
1

# 1.2 Features

  • Control flow with <% %>
  • Escaped output with <%= %> (escape function configurable)
  • Unescaped raw output with <%- %>
  • Newline-trim mode ('newline slurping') with -%> ending tag
  • Whitespace-trim mode (slurp all whitespace) for control flow with <%_ _%>
  • Custom delimiters (e.g. [? ?] instead of <% %>)

# 1.3 Basic usage

// Set ejs as the default template engine
app.set("view engine", "ejs");
//Configure the template path
app.set("views", path.resolve(__dirname, "views"));
//Render a template engine and return it to the browser
res.render("students", { name });

💡Note that the template engine needs to be rendered by express before it can be used

  • res.render() is used to render a template engine and return it to the browser
  • An object can be passed as the second argument to render so that the data in the object can be accessed in the template
    • res.render("students", { name: "AAAAAAA", age: 18, gender: "Male" })
  • <%= %> When outputting content in ejs, it automatically escapes special symbols in the string; (This is mainly designed to avoid xss attacks)

# 1.4 Tags

  • <% 'Scriptlet' tag, for control-flow, no output
  • <%_ 'Whitespace Slurping' Scriptlet tag, strips all whitespace before it
  • <%= Outputs the value into the template (escaped)
  • <%- Outputs the unescaped value into the template
  • <%# Comment tag, no execution, no output
  • <%% Outputs a literal <%
  • %%> Outputs a literal%>
  • %> Plain ending tag
  • -%> Trim-mode ('newline slurp') tag, trims following newline
  • _%> 'Whitespace Slurping' ending tag, removes all whitespace after it

# cors middleware to solve cross-domain

  • Install middleware: npm install cors
  • Import middleware: const cors = require('cors')
  • Configure middleware: app.use(cors())

# CORS

  • CORS (Cross-Origin Resource Sharing) solves cross-domain by using HTTP response headers to determine whether the browser prevents front-end JS code from accessing resources across domains
  • By default, the browser's same-origin security policy prevents web pages from accessing resources "across domains". However, if the interface server is configured with CORS-related HTTP response headers, the cross-domain access restrictions on the browser side can be lifted
  • CORS is primarily configured on the server side. The client browser can request a CORS-enabled interface without any additional configuration.
  • CORS has browser compatibility. Only browsers that support XMLHttpRequest Level 2 can properly access the server-side interface with CORS enabled (e.g. IE10+, Chrome4+, FireFox3.5+).

# CORS common response headers

  • ``Access-Control-Allow-Origin`'': the URL of the foreign domain where access to the resource is allowed
res.setHeader("Access-Control-Allow-Origin", "http://bruceblog.io");
res.setHeader("Access-Control-Allow-Origin", "*");
1
2
  • Access-Control-Allow-Headers
  • By default, CORS only supports the following 9 request headers sent by the client to the server: Accept, Accept-Language, Content-Language, DPR, Downlink, Save-Data, Viewport-Width, Width, Content-Type ( values are limited to one of text/plain, multipart/form-data, application/x-www-form-urlencoded)
  • If the client sends additional request headers to the server, the additional request headers need to be declared on the server side, via Access-Control-Allow-Headers, otherwise this request will fail!
res.setHeader("Access-Control-Allow-Headers", "Content-Type, X-Custom-Header");
1
  • Access-Control-Allow-Methods
  • By default, CORS only supports clients to initiate GET, POST, HEAD requests. If the client wishes to request resources from the server via PUT, DELETE, etc., it needs to specify the HTTP methods allowed for the actual request via ``Access-Control-Allow-Methods`'' on the server side
res.setHeader("Access-Control-Allow-Methods", "POST, GET, DELETE, HEAD");
res.setHEader("Access-Control-Allow-Methods", "*");
1
2

# CORS request classification

# Simple request

  • Request method: GET, POST, HEAD One of the three
  • HTTP header information no more than the following fields: No custom header fields, Accept, Accept-Language, Content-Language, DPR, Downlink, Save-Data, Viewport-Width, Width, Content-Type (only three values) application/x-www-formurlencoded, multipart/form-data, text/plain)

# Preflight request

  • Request Method type other than GET, POST, HEAD
  • Request header contains custom header fields
  • Data in application/json format is sent to the server

Before the browser formally communicates with the server, the browser sends an OPTION request for pre-testing to see if the server will allow the actual request, so this OPTION request is called a "pre-test request". After the server successfully responds to the preflight request, the real request is sent with the real datafejs

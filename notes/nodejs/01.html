# ⛳️01. Introduction to nodejs

# 1. What is Node.js?

Node.js is an open-source and cross-platform JavaScript runtime environment.

Node.js® is a JavaScript runtime built on Chrome's V8. JavaScript engine outside of the browser.

This allows Node.js to be very performant. A Node.js app runs in a single process, without creating a new thread for every request. Node.js provides a set of asynchronous I/O primitives in its standard library that prevent JavaScript code from blocking and generally, libraries in Node.js are written using non-blocking paradigms, making blocking behavior the exception rather than the norm.

When Node.js performs an I/O operation, like reading from the network, accessing a database or the filesystem, instead of blocking the thread and wasting CPU cycles waiting, Node.js will resume the operations when the response comes back.

This allows Node.js to handle thousands of concurrent connections with a single server without introducing the burden of managing thread concurrency, which could be a significant source of bugs.

Node.js has a unique advantage because millions of frontend developers that write JavaScript for the browser are now able to write the server-side code in addition to the client-side code without the need to learn a completely different language.

In Node.js the new ECMAScript standards can be used without problems, as you don't have to wait for all your users to update their browsers - you are in charge of deciding which ECMAScript version to use by changing the Node.js version, and you can also enable specific experimental features by running Node.js with flags.

# 2. Composition of Node.js

# 2.1 Differences between Node.js and the Browser

Both use the same JS engine. Writing JS in Node.js is almost no different from writing JS on the front end. The difference is that Node.js has no browser or page tag related APIs, but has some new Node.js related APIs. in layman's terms, for developers, writing JS on the front end is used to control the browser; writing JS in a Node.js environment can control the entire computer.

As we know, JavaScript is composed of three parts.

  • ECMAScript
  • DOM: tag element related API
  • BOM: Browser-related APIs

ECMAScript is the syntax of JS; DOM and BOM are the APIs provided by the browser side for JS.

And the composition of Node.js is divided into:

  • ECMAScrip: all syntax of ECMAScript is available in the Node environment.
  • Some additional APIs provided by the Node environment (including file, network, and other related APIs).

# 2.3 Summary

  • Node is a server-side JavaScript interpreter
  • Node.js is a JavaScript runtime environment based on the Chrome V8 engine
  • Node.js uses an event-driven, non-blocking I/O model, making it lightweight and efficient
  • Node.js's package manager, npm, is the world's largest open source library ecosystem
  • Node.js is a dynamic language that runs as a server-side Javascript

# 3. Node.js Applications

# 3.1 The BFF Pattern (Backend for Frontend)

Under the BFF model, the overall division of labor is clear, ** the back-end is responsible for service implementation through languages such as Java/C++, ideally providing the front-end with an RPC interface based on the domain model, while the front-end directly calls the server-side RPC interface at the BFF layer to get data**, process and consume data on demand, and realize human-computer interaction. The BFF model-based R&D is very suitable for full-stack engineers with front-end technical background. The benefits of this model are obvious: the back-end can focus on the business domain and think more from the perspective of the domain model, while the data from the page perspective is left to the front-end full-stack engineers to handle. **Domain model and page data are two modes of thinking, which can be well decoupled through BFF, making each other more professional and efficient.

The BFF will do the following.

  • Call the relevant microservices APIs and obtain the needed data
  • Format the data based on the frontend representation
  • Send the formatted data to the frontend

In Web services, an intermediate layer is built, the front-end accesses the interface of the intermediate layer, and the intermediate layer then accesses the Java/C++ services in the back-end. This type of service does not require much server computing power, but requires a high degree of program flexibility. These two characteristics coincide with the advantages of Node.js, which is well suited for the BFF layer and has the following advantages.

  • For the front-end: giving the front-end** the ability to freely assemble back-end data**, which reduces a lot of business communication costs and speeds up business iteration; and, front-end students are able to autonomously decide how to communicate with the back-end.
  • For the backend and Ops, the benefits are: security (not exposing the main server to the outside), reducing the complexity of the main server, etc.

👀Reference: the benefits of using the BFF pattern

# 3.2 Server-side rendering

CSR / Client side render: The front-end requests data through a bunch of interfaces, and then dynamically processes and generates the page structure and presentation through JS. The advantages are separation of front and back ends, reduced server pressure, and partial refresh. Disadvantages are bad for SEO (if your page then fetches content asynchronously via Ajax, the crawler tool doesn't wait for the asynchrony to complete before crawling the page content), and slow first screen rendering.

SSR / Server Side Render: Instead of interface data, the server returns an entire page (or entire floor) of HTML strings that the browser can display directly. In other words, it is rendered directly on the server side, and then packaged and returned to the front-end in one go. The advantages are good for SEO and fast first screen rendering.

Summary: Search Engine Optimization + First Screen Speed Optimization = Server Side Rendering

👀Reference: SSR vs CSR Node.js Server Side Rendering (SSR) using EJS

# 3.3 Build Express, Koa Framework based small web backend with nodejs

Nowadays, many companies are using Node.js to develop interfaces for their backend management systems, which, after all, do not have high performance and concurrency requirements. With Node.js, you can directly manipulate the DB, add, delete, and generate interfaces through JS, which greatly reduces the learning threshold of the front-end.

# 4. Characteristics of Node.js

There are a number of characteristics that make Node.js what it is:

  • Google Chrome V8 JavaScript Engine: This runtime environment is built on the Google Chrome V8 JavaScript runtime engine. In the same way a Java Virtual Machine translates bytecode, the Chrome V8 JavaScript engine takes JavaScript and makes it readable.
  • Modules/Packages: Node.js has npm, a node package manager, with a library of over 350,000 packages to help get your project or application off the ground with efficiency and ease.
  • Event Driven, Single-Threaded I/O Model: JavaScript relies on user interactions or events to run. In most cases, code is run synchronously. Server requests and other such asynchronous tasks rely on a system of promises or async/await functions to handle these inputs and outputs.
  • Node.js uses an asynchronous, non-blocking model. By "single threaded", we mean that there is only one main thread in Node. To ensure that the main thread is not blocking, it is used to receive client requests. But it does not handle specific tasks. Instead, Node has a thread pool behind it, which handles long-running tasks (e.g. IO operations, network operations). The tasks in the thread pool are executed through a queue and event loop mechanism.
